This is a short document describing the preferred coding style. Coding style is very personal, but this is what goes for anything that I have to be able to maintain, and I’d prefer it for most other things too. Please at least consider the points made here.

Indentation

As a rule, it is better to use space instead of "tab", because it allows you to make indentation configurating more flexible.

  • Block indentation: +2 spaces:

    • Each time a new block or block-like construct is opened, the indent increases by two spaces. When the block ends, the indent returns to the previous indent level. The indent level applies to both code and comments throughout the block.

    
    class InnerClass {
        constructor() {}

        /** @param {number} foo */
        method(foo) {
            if (condition(foo)) {
                try {
                    // Note: this might fail.
                    something();
                } catch (err) {
                    recover();
                }
            }
        }
    }
  • Vertical whitespace:

    • A single blank line appears within method bodies and between consecutive methods in a class or object literal, sparingly to create logical groupings of statements. Blank lines at the start or end of a function body are not allowed.

    In the example, the initialization of the variables, the main loop and the return of the result are separated:

    
    function pow(x, n) {
        var result = 1;

        if (var i = 0; i > n; i--) {
        result *= x;
        }

        return result;
    }
Line Length

There’s no line length limit in javascript although 80 and 120 characters is a guideline. Any line that would exceed this limit must be line-wrapped. Line-wrapping is defined as breaking a single expression into multiple lines. There is no comprehensive, deterministic formula showing exactly how to line-wrap in every situation. The prime directive of line-wrapping is: prefer to break at a higher syntactic level. Preferred:


    currentEstimate =
    calc(currentEstimate + x * currentEstimate) /
    2.0f;}

In the preceding example, the syntactic levels from highest to lowest are as follows: assignment, division, function call, parameters, number constant.

Braces

Braces are required for all control structures (i.e. if, else, for, do, while, as well as any others), even if the body contains only a single statement.

  • Nonempty blocks:

    • Braces follow the Kernighan and Ritchie style (Egyptian brackets) for nonempty blocks and block-like construct:

    • No line break before opening brace.
    • Line break after the opening brace.
    • Line break before the closing brace.
    
    class InnerClass {
        constructor() {}

        /** @param {number} foo */
        method(foo) {
        if (condition(foo)) {
            try {
            // Note: this might fail.
                something();
            } catch (err) {
                recover();
            }
        }
        }
    }
  • Empty blocks:

    • An empty block or block-like construct may be closed immediately after it is opened, with no characters, space, or line break in between:

    
    function doNothing() {}
Naming

Identifiers use only ASCII letters and digits, and, in a small number of cases noted below, underscores and very rarely dollar signs.

Rules by identifier type:

  • Class names:

    • Classnames are written in UpperCamelCase. Type names are typically nouns or noun phrases. For example, Request, ImmutableList, or VisibilityMode.

  • Method names:

    • Method names are written in lowerCamelCase.Method names are typically verbs or verb phrases. For example, sendMessage or stop_.

  • Constant names:

    • Constant names use CONSTANT_CASE: all uppercase letters, with words separated by underscores.

  • Parameter and Local variable names:

    • Parameterand Local variable names are written in lowerCamelCase.

Functions:

There are two ways to position the functions required to execute code:

  • Functions over the code that uses them:
    • 
    function createElement() {
    ...
    }

    function setHandler(elem) {
    ...
    }

    function walkAround() {
    ...
    }


    var elem = createElement();
    setHandler(elem);
    walkAround();
  • Functions under the code that uses them:
    • 
    var elem = createElement();
    setHandler(elem);
    walkAround();



    function createElement() {
    ...
    }

    function setHandler(elem) {
    ...
    }

    function walkAround() {
    ...
    }

As a rule, it is better to place functions under the code that uses them. The fact is that when reading the code, we want to know first of all what it does

Comments:

Comments are indented at the same level as the surrounding code. They may be in /* … */ or //-style. For multi-line /* … */ comments, subsequent lines must start with * aligned with the * on the previous line, to make comments obvious with no extra context. “Parameter name” comments should appear after values whenever the value and method name do not sufficiently convey the meaning.


    /*
    * This is
    * okay.
    */

    // And so
    // is this.

    /* This is fine, too. */

    someFunction(obviousParam, true /* shouldRender */, 'hello' /* name */);

So what comments are useful and welcome?

First of all, there should be a minimum of comments that answer the question "what happens in the code?". Your code should be clear. Useful comments:

  • Architectural commentary – " how it works, arranged in general."

    • What and why this script. What components , what technologies are used and their interactions. These comments are especially needed if the project is large.

  • A reference comment before the function is about what it does, what parameters it accepts, and what it returns.

    /**
    * Returns x to the power of n, only for natural n
    *
    * @param {number} x   Number for exponentiation.
    * @param {number} n   the exponent, a natural number.
    * @return {number}    x to the power of n.
    */
    function pow(x, n) {
    ...
    }
